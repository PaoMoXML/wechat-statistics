# 微信消息统计工具 · 技术方案

## 一、方案概览

| 项 | 决策 |
|---|---|
| 目标客户端 | Windows 微信 4.x |
| 数据源 | 本地 `xwechat_files` 加密数据库（SQLCipher 4） |
| 数据获取 | **从微信进程内存提取密钥 → 离线解密数据库** |
| 统计维度 | 联系人/群消息量、活跃时段、字数词频、媒体统计（全选） |
| 输出 | 本地网页仪表盘（HTML + ECharts，离线可用） |
| 语言/形态 | Rust，单二进制 CLI |

**核心架构决策——分两层推进，把最难的逆向和最高价值的统计解耦：**

> **Phase 1（推荐先做）**：不自己写密钥提取/解密，直接复用成熟的 Rust 工具 `wechat-dump-rs`（或 Go 的 `chatlog`）产出**已解密的明文 SQLite**，本工具专注「解析 + 统计 + 仪表盘」。这样能在 1–2 周内端到端跑通，规避逆向风险。
> **Phase 2（自包含）**：再把密钥提取与 SQLCipher 解密做进自己的二进制，成为单一可执行文件。

这样设计的好处是：逆向（强依赖微信版本、最容易失效）和业务（统计/可视化、相对稳定）分离，即使微信更新导致密钥提取失效，统计/渲染层也完全不受影响。

---

## 二、数据获取方案（核心难点，详细展开）

WeChat 4.x 的核心结论（已核实）：本地数据库存在 `xwechat_files` 目录，使用 **SQLCipher 4 (AES-256)** 加密；解密密钥会**缓存在运行中的微信进程内存里**，且**在大版本内稳定不变**（提取一次即可长期使用）——这是整个工具可行的关键前提。

整体数据流：

```
微信运行中进程内存
      │  [密钥提取] 进程内存特征扫描
      ▼
  64-hex DB Key（大版本内稳定，缓存即可）
      │  [SQLCipher 4 解密]
      ▼
明文 SQLite：message_0.db … / contact.db / session.db / media.db
      │  [Schema 映射层]（按版本适配）
      ▼
   统一数据模型 Message / Conversation / Contact
      │  [统计引擎]
      ▼
   统计结果 JSON
      │  [渲染层]
      ▼
   dashboard.html（内嵌 ECharts）
```

**为什么这个方案可行/可靠：**
- 密钥在大版本内不变，**提取一次缓存到本地配置**，后续无需微信在线也能离线解密（cloudbak 文档）。
- 密钥提取的成熟实现已开源（`wechat-decrypt`、`wechat-dump-rs`、`chatlog`、`wx_key`），无需从零逆向；看雪的分析帖与 Sarv's Blog 提供了完整原理。
- 解密后得到的就是标准 SQLite，可用常规 SQL 直接查询聚合。

**Phase 1 落地方式（务实）：** 直接调用 `wechat-dump-rs` 一键导出密钥并解密所有库，本工具读它产出的明文 `.db`。

**Phase 2 原生实现：** 用 `windows` crate 的 `ReadProcessMemory` 扫描微信进程内存定位密钥，再用 `rusqlite` 的 `bundled-sqlcipher` 特性以 `PRAGMA key = "x'<hex>'"` 直接打开加密库（或纯 Rust 用 `aes`/`cbc`/`pbkdf2`/`hmac`/`sha2` 逐页解密成明文库）。

## 三、模块设计（Rust crate 划分）

```
src/
├── main.rs              // clap CLI，编排整条流水线
├── key_extract.rs       // [Phase2] 进程内存扫描提取密钥
├── decrypt.rs           // [Phase2] SQLCipher 4 解密
├── loader.rs            // 打开明文 SQLite（Phase1 直接读，Phase2 读内存解密结果）
├── schema.rs            // ★ 版本适配层：探测表结构 → 映射到统一模型
├── model.rs             // Message / Conversation / Contact / MediaType
├── stats/
│   ├── mod.rs
│   ├── volume.rs        // 维度1：联系人/群消息量 + 时间跨度
│   ├── temporal.rs      // 维度2：小时/星期/月份活跃分布
│   ├── lexical.rs       // 维度3：字数 + jieba 词频 + Top-N
│   └── media.rs         // 维度4：图片/语音/视频/文件计数
├── render.rs            // askama 模板 → dashboard.html
└── templates/
    └── dashboard.html   // 内嵌 vendored echarts.min.js
```

**最关键的模块是 `schema.rs`**：WeChat 4.x 表/列名与 3.x 完全不同，且会随小版本变化。绝不能硬编码列名——要**先 `PRAGMA` 探测实际表结构，再用适配层映射**到统一模型。这是规避「微信一更新就崩」的核心防线。

## 四、四个统计维度的实现要点

| 维度 | 实现要点 |
|---|---|
| **联系人/群消息量** | SQL `GROUP BY conversation` 聚合计数；附首条/末条时间、最近活跃。**优先 SQL 聚合**而非全量载入内存 |
| **活跃时段分布** | `mesCreateTime`（Unix 秒，需核对时区）→ `chrono` 转本地时区 → 按小时/星期/月份分桶，SQL `GROUP BY` 出直方图 |
| **字数与词频** | **仅对文本类消息**统计；中文用 `jieba-rs` 分词；剔除媒体/链接的 XML 内容；产出总字数、人均字数、Top-N 高频词、词云数据 |
| **媒体统计** | 按 `mesType` 分类计数。参考类型码（**需对照实际库核对**）：图片=3、语音=34、视频=43、文件/应用=49、表情=47、系统=10000 |

**性能策略**：库可能达 GB 级。能用 SQL 聚合的绝不拉进 Rust；必须全量遍历的（如词频）用 `rayon` 并行 + 流式读取，结果物化为紧凑中间 JSON。

## 五、网页仪表盘

- **形态**：单个自包含 `dashboard.html` + 内嵌 `data.js`（统计结果 JSON），双击即可在浏览器打开。
- **图表库**：ECharts（`echarts.min.js` 打包进项目，**离线可用**，无外部依赖）。
- **图表规划**：
  - 联系人/群消息量 **Top 排行榜**（横向柱状图）
  - 24 小时 / 星期 / 月度 **活跃热力图与折线**

- **渲染**：`askama` 编译期模板，Rust 端把各维度结果注入。Phase 3 可加 `axum` 起本地服务做交互筛选。

## 七、风险与对策

| 风险 | 对策 |
|---|---|
| **Schema 随版本变化**（最大风险） | `schema.rs` 运行时探测表结构 + 适配层，**禁止硬编码列名**；Phase 0 先 dump 一份真实库定锚 |
| 密钥提取随微信更新失效 | Phase 1 复用社区工具（`wechat-dump-rs`/`chatlog`）；密钥大版本内稳定，提取一次缓存 |
| 库体量巨大（GB 级） | SQL 聚合优先、`rayon` 并行、流式读取、结果物化 |
| 内容字段含 XML（媒体/链接） | 按 `mesType` 过滤，文本类才分词；`quick-xml` 抽取媒体摘要 |
| 隐私敏感 | **全本地、不联网**；解密产物由用户掌控并加告警；不写日志泄露内容 |

## 十、Phase 0 落地记录

### 已完成
- 项目骨架：`Cargo.toml`（rusqlite bundled / clap / anyhow / serde / serde_json / chrono）
- schema 探测程序 `src/probe.rs` + CLI `src/main.rs`，提供两个子命令：
  - `schema`：探测已解密库的表结构（列类型/PK/NOTNULL、行数、采样行、**时间戳语义猜测**）
  - `selftest`：构造仿微信库并自检（本机无微信/无 sqlite3 CLI 也能验证流程）
- 已验证：编译通过、selftest 端到端跑通、JSON 导出正常、未解密库给出明确错误提示

### 用法
```bash
# 1) 自检（无需微信数据）
cargo run -- selftest

# 2) 探测真实已解密库（文件或目录）
cargo run -- schema --db /path/to/decrypted_xwechat/message_0.db
cargo run -- schema --db /path/to/decrypted_xwechat/        # 探测目录下所有 .db
cargo run -- schema --db <PATH> --no-count                  # 大库跳过 COUNT(*)
cargo run -- schema --db <PATH> --no-samples                # 只要结构、不要采样，JSON 最小
cargo run -- schema --db <PATH> --json probe.json           # 同时导出机器可读结果
```

> 💡 **体积优化**：微信会为每个会话/联系人单独建一张结构相同的消息表，一个账号可达数千张。
> 探测程序按「表结构指纹」自动分组去重——结构相同的表合并成一项，只对代表表采样/计数，
> 因此几千张重复表只会坍缩成一条，JSON 体积与探测耗时都大幅下降。
> 若只要结构清单，加 `--no-samples` 可进一步缩小。

> ⚠️ 真实库仍处于 SQLCipher 加密状态，需先用 [wechat-dump-rs](https://github.com/0xlane/wechat-dump-rs) / [chatlog](https://github.com/sjzar/chatlog) 解密后再探测。

### 已确认 schema（基于真实账号探测，Phase 0 完成）

24 个 db 文件的角色已厘清，核心映射链：

```
会话 username ──(session.db.session_last_message: db_stem + table_name)──▶
  message_{db_stem}.db 的 Msg_{table_name} 表 ──(real_sender_id)──▶
  contact.db.contact.id → username/nick_name
```

**① 消息表**（`message_0~4.db` / `biz_message_0~4.db` 的 `Msg_<32hex>`，每会话一张，结构一致）：

| 列 | 用途 |
|---|---|
| `local_id` (PK) | 本地主键 |
| `local_type` | ★ **打包类型** `(subtype<<32)|base_type`，统计取低 32 位 |
| `create_time` | ★ Unix 秒（已确认） |
| `real_sender_id` | ★ **Name2Id.rowid 外键**（不是 `contact.id`！见下「关键纠正」） |
| `message_content` | ★ 正文，但 **zstd 压缩**（魔数 `28 b5 2f fd`），文本/媒体均压缩 |
| `sort_seq` | ★ 会话内时序（回复时延/每日首发按它排序最准） |
| `server_id` / `status` / `compress_content` / `packed_info_data` / `WCDB_CT_*` | 辅助字段 |

**② 联系人表**（`contact.db.contact`，~6889 行）：`id`(PK)、`username`(wxid/群ID)、`nick_name`、`remark`、`alias`、`is_in_chat_room`。

**③ 会话映射**（`session.db.session_last_message`，~455 行）：`username` → `db_stem` + `table_name`。

**关键发现**：`local_type` 不是单一类型码，而是 `(subtype << 32) | base_type` 打包值
（如 `21474836529 = (5<<32)|49` → 文件/应用消息）。统计必须取低 32 位基础类型，
否则同一类消息会被拆成几十个 subtype 变体。已封装为 `schema::base_type()`。
常见基础类型：1=文本、3=图片、34=语音、42=名片、43=视频、47=表情、48=位置、49=文件/应用、50=音视频通话、10000=系统。

> **⚠️ 关键纠正（`real_sender_id`）**：Phase 0 曾误判 `real_sender_id` = 发送者的 `contact.id`，
> 导致 dig/stats 的「我 vs 对方」全部失真（症状：1v1 会话里 sender 全显示成 gh_* 公众号——因为
> Name2Id.rowid 的小整数空间与 contact.id 偶然重合）。真实语义（参考 [/home/xmlin/rust/WeChatDataAnalysis](../WeChatDataAnalysis) + 真实库验证）：
> - `real_sender_id` 是 **`Name2Id` 表的隐式 `rowid` 外键**，取真实 wxid 要 `LEFT JOIN Name2Id n ON real_sender_id = n.rowid` → `n.user_name`。
> - **每个 `message_*.db` 都有自己的 Name2Id**，rowid 跨库不同（实测 message_0 里我=rowid 7，message_1 里=rowid 18，biz_message_0 里=rowid 2）。
> - 「我自己」= `SELECT rowid FROM Name2Id WHERE user_name = <账号目录名>`（per-DB），`real_sender_id == 该 rowid` 即「我发的」。已实现为 `loader::self_rowid_for`。
> - `message_content` / `compress_content` 是 **zstd 压缩**；群聊真实发送者还在解压后的文本（`"wxid:\n正文"` 前缀）或 XML `<fromusername>` 里（本工具目前不读正文，按 rowid 区分收发已够用）。
> - Name2Id 列名兼容：message 用 `user_name`，部分 biz 库用 `username`。

### 验证结果（真实库，只读）

`inspect --data <root>` 端到端跑通：解析 6889 联系人 / 455 会话 / ~10.2 万条消息，
Top 会话时间跨度 2025-06 ~ 2026-06 正确，类型分布合理（文本占多数、表情/图片次之）。
全程未读取正文内容。

## 十一、Phase 1 落地记录

### 已完成

统计引擎 + 五大能力（消息量 / 活跃时段 / 单会话深挖 / 情侣报告 / 词频），真实账号验证通过。`cargo test` 17/17。

**① 引擎与基础维度**
- **统计引擎架构**：每个维度做成「在线累加器」（`observe_table` 喂入 → `finalize` 出结果），
  多维度**共用单次遍历**，避免对海量消息表反复扫描。
- `loader::for_each_conversation`：按 `db_stem` 分组、每个 `message_*.db` 只开一次连接，
  单表出错只警告跳过不中断；并按库解析 per-DB 的 `self_rowid` 一并传入闭包。
- **维度1 · 消息量**（`stats/volume.rs`）：总会话 / 总消息 / 基础类型分布 / Top-N 会话。
- **维度2 · 活跃时段**（`stats/temporal.rs`）：小时（24）/ 星期（周一为起点）/ 月份分布，
  全部用 SQLite `strftime` + `'unixepoch','localtime'` 在库内聚合，时间戳不进内存。
- `stats` 子命令：文本报告（千分位、等比柱状图、**媒体拆分**）+ 可选 `--json`。

**② 单会话深挖 `dig`**（`src/dig.rs`）
- 发送比例（我 vs 对方）、类型分布、回复时延（中位/均值/p90）、每日首发。
- 联系人昵称解析：`ContactIndex` 把 wxid 映射到 `nick_name`/`remark`，报告直接显示昵称。
- 定位方式：`--with <昵称片段>` / `--rank N` / `--user <wxid>`。

**③ 情侣报告 `couple`**（`src/couple.rs`）— 时序叙事，纯计算可单测
- 在一起第 N 天 / 活跃天数占比 / 最长连续（含当前连续）/ **最长沉默（含时间段）** / 最嗨一天 / 一起熬过的夜（0–4 点）/ 通话次数（base_type=50）。
- 秒回率（30s / 1min 内，我 vs 对方）。
- 跨月/跨年的连续/沉默用 chrono `NaiveDate::num_days_from_ce` 精确计算。

**④ 正文解码 `content`**（`src/content.rs`）
- `message_content` / `compress_content` 的 **zstd 解压**（魔数 `28 b5 2f fd`），用 `get_ref` 取原始字节避免 TEXT affinity 破坏二进制 → UTF-8 + 去群前缀。
- 按需读正文：只有文本分析（couple）读，统计骨架/dig 从不碰正文。
- 文本指标：总字数（我/对方）、最长一条、关键词计数（默认情侣词表，我 vs 对方）。

**⑤ 词频 `lexical`**（`src/lexical.rs`）
- `jieba-rs` 中文分词 + 内置 ~150 词停用词表 + 数字/标点/单字噪音过滤。
- 总体 / 我 / 对方 三组高频词（`WordFreq`），jieba 用 `OnceLock` 全局只初始化一次。

### 关键决策

| 决策 | 理由 |
|---|---|
| 类型取基础值在 **SQL 侧** `local_type & 4294967295` | 比逐行调 Rust 的 `base_type()` 快；避免 subtype 把同类消息拆成几十桶 |
| 时段聚合全部 SQLite `strftime` + `localtime` | 库内分桶，只回传 24/7/N 个聚合行，正文/时间戳从不进 Rust |
| **媒体统计内嵌在消息量报告** | 维度4 的基础计数（图片/语音/视频/表情/文件/通话）已随类型分布得出，无需单独遍历 |
| 在线累加器（非全量载入） | 维度1+维度2 共用单次遍历；将来加词频维度也能挂进同一次遍历 |

### 用法

```bash
# 全局：消息量 + 活跃时段 + 媒体拆分
cargo run --release -- stats --data <root> --top 15 [--json out.json]

# 单会话深挖
cargo run --release -- dig --data <root> --with <昵称片段> [--rank N | --user <wxid>]

# 情侣报告（时序叙事 + 文本指标 + 高频词，可导出 HTML）
cargo run --release -- couple --data <root> --with <昵称> [--words 12] [--html report.html]
```

### 验证（真实库，102,126 条消息）

三个交叉校验全部自洽，证明逻辑正确：
1. 文本 53,511 + 媒体 46,545 + 系统 1,942 + 其它 128 = **102,126** = 消息总数 ✓
2. 活跃时段总计 = **102,126**（两个累加器在同一遍历里精确计数同一批行）✓
3. 单测覆盖 `local_type` 掩码（纯 `1` + 打包 `(5<<32)|1` 合并为 base `1`）✓

真实库还暴露一处数据特征：`session_last_message` 有 46 行 `db_stem` 为空，
已干净跳过（非错误）。凌晨 0 点的小峰值、周六骤降等异常分布，留待 Phase 2 单会话深挖。
